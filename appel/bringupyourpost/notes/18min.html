PU's

At the end of every Appel level a clone of that Appel level is created, but shifted down one tile. You can't see this clone because of the camera stops, but the player can enter it. Interestingly there is actually an infinite amount of PU's, each PU to the right of the other and down one tile. Additionally there is an infinite amount of PU's going the left, and each PU's goes up one tile. Although if you try to enter those PU's, the code will push you out. But interestingly due to the order code is run you can actually wall jump off of a tile from the PU and even jump on it. So why does this happen? Every tile in an Appel level is stored in a list. The bottom left tile of the level is stored in index 1 of the list, the tile just right of that is stored in index 2 of the list. This continues until you get the bottom right tile. Then the next tile is the one just above the first tile and then this pattern repeats. As you can see going up increases the index by the width of the level. But going to the right increases the index by 1. So what if we take the bottom right most tile and figure out the one to right of it. Here as an example let's assume the width is 196. This means the right most tile must have index 196. So the one to the right of it must have index 197. But if we look at the first tile it has index 1. So the one above it must have index 197. As you can see this tile has the same tile index as the tile 196 units to the right and 1 tile down. This means they have the exact same index. Additionally every tile has the same tile index as the tile 196 units to the right and 1 tile down. So because they have the same tile index, they also refer to the same tile which means, they will be the exact same. But PU's are not necessarily the same. A lot of entities are not created in PUs. The process for creating the entities involves going through the tile list and for each tile figuring out what location has that tile index. There are many tiles which have that tile index, but it picks the tile that is not in a PU. Keys cannot be collected in PU's, neither can red apples or yellow apples that are attached to moving platforms. A player can collect a flag in a PU but cannot die because the camera is not positioned near the player. Spikes only work when on screen. Keydoors and crumble blocks can be broken in PU's but they affect every PU when broken. Conveyor belts are also affected by PU's. Trampolines are really interesting they make the BOING sound effect when you touch them in a PU, but trampolines check if the player is near the non-PU location to check if a bounce should occur, so trampolines just don't work. Checkpoints work in PU's but you will respawn in the non PU. Collecting the finish flag will also work. Speaking of finish flags... let's talk about Level 0.

Overshadowing

Look at this setup. Notice how for some reason Appel stops riding PLATFORM A when Appel also lands on PLATFORM B. This is called overshadowing. See PLATFORM B is stored after PLATFORM A in the objects list becaues PLATFORM B has a larger index the PLATFORM A. This leads to something pretty interesting, see every single frame Appel goes through every dynamic and checks if he is touching that moving platform and if so how to modify the speed. So what happens in this situation is that we first check PLATFORM A, and we see that we are touching and so we modify the speed to make appel follow PLATFORM A. Then we check if we are touching PLATFORM B, and we see that we are and so we modify the speed to make appel follow PLATFORM B which overrides PLATFORM A. I call this quirk Dynamic Overshadowing because it seems that PLATFORM A is completely overshadowed by PLATFORM B.

Floating Point stuff

Did you know that if you open an appel level and then move a little bit to the start the timer and then wait a little less then 10 million years the timer will stop. This is because of floating point imprecision. see the timer on screen is computed via a variable called time. This variable starts incrementing after you start moving. It increments onces a frame so 30 times a second. Now after waiting 10 million years, the frame variable reaches a very special value: 9,007,199,254,740,992 or 2^53. Now scratch is written in JS and numbers are stored in JS as 64 bit floating point numbers. Floating point numbers become less and less dense the more and more you go up. So the distance between 1 and the floating point number immediately after one is only 1 part in 4.5 quadrillion. This gap doubles at every power of two, other stuff happens near zero and infinity, but that is irrelevant. The gap at one starts at 2^-52, and so naturally by the time you make it to 2^53, the gap should be 2. The next floating point number after 2^53 is 2^53 + 2. So if you add 1 to 2^53 you have to make a choice between 2^53 and 2^53 + 2. The way Javascript handles floats is to round towards closest. But we need a tie breaker. So what Javascript does is that it rounds to the number which is more 2-ish. 2^53 + 2 is even but it isn't divisible by 4. Whilst 2^53 is. So when we increment tick it rounds back down to its previous value. Thus fixing the tick variable. Interestingly if Javascript rounded away from zero instead, this problem would be delayed by another 5 million years.
